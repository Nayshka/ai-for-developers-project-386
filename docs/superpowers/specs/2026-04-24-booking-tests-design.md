# Дизайн тестового покрытия сценариев бронирования

## Контекст

Проект реализует учебное приложение «Запись на звонок» с гостевыми сценариями бронирования и разделом владельца. Backend построен на Fastify, TypeScript и Vitest. Frontend построен на React, Vite, Mantine, React Router и TanStack Query. Данные хранятся в памяти процесса, поэтому тесты могут запускаться изолированно без внешней базы данных.

Существующие материалы:

- `docs/test-scenarios.md` фиксирует основные пользовательские сценарии.
- `server/src/__tests__/backend.test.ts` уже проверяет часть backend API через `app.inject()`.
- `.github/workflows/hexlet-check.yml` является служебным autogenerated workflow и не должен изменяться.

## Цели

1. Зафиксировать основные пользовательские сценарии проверки бронирования.
2. Подготовить интеграционные тесты для backend-правил бронирования.
3. Добавить Playwright e2e-тесты для ключевых пользовательских потоков через браузер.
4. Настроить отдельный CI workflow для запуска build, backend integration tests и browser e2e tests.

## Не цели

- Не добавлять авторизацию, роли пользователей или внешние календарные интеграции.
- Не менять бизнес-логику бронирования вне требований, необходимых для тестируемости.
- Не заменять существующие Vitest-тесты браузерными тестами.
- Не редактировать autogenerated Hexlet workflow.

## Подход

Покрытие делится на два уровня:

- Backend integration tests проверяют API-контракты, бизнес-правила, временные ограничения, конфликты слотов и валидацию. Эти тесты быстрые, детерминированные и используют текущий `buildApp({ now })`.
- Playwright e2e tests проверяют, что пользователь может пройти основные потоки через реальный UI: выбрать встречу, забронировать слот, увидеть подтверждение, открыть список владельца и создать новый тип встречи.

Рекомендуемый набор реализуется как баланс между стабильностью и пользовательской достоверностью: сложные edge cases остаются на backend-уровне, а браузерные тесты покрывают только критичные happy path и один важный error path.

## Пользовательские сценарии

### Сценарий 1. Гость видит типы встреч

Гость открывает главную страницу и видит seed-типы встреч `consultation` и `demo` с названием, описанием и длительностью. Сценарий покрывается backend-тестом `GET /event-types` и Playwright e2e-проверкой главной страницы.

### Сценарий 2. Гость выбирает тип встречи и видит слоты

Гость открывает страницу конкретного типа встречи и видит доступные слоты в пределах рабочего окна 09:00-19:00 на ближайшие 14 дней. Сценарий покрывается backend-тестом `GET /event-types/:id/slots` и Playwright e2e-переходом к странице бронирования.

### Сценарий 3. Гость успешно бронирует слот

Гость выбирает свободный слот, вводит имя и email, отправляет форму и видит подтверждение. Backend создает бронирование со статусом `confirmed`. Сценарий покрывается backend-тестом `POST /bookings` и Playwright e2e happy path.

### Сценарий 4. Владелец видит созданное бронирование

После успешного бронирования владелец открывает `/owner/bookings` и видит имя гостя, email, тип встречи, дату, время и статус. Сценарий покрывается backend-тестом `GET /owner/bookings` и тем же Playwright e2e happy path.

### Сценарий 5. Повторное бронирование занятого слота запрещено

Если слот уже забронирован, повторный запрос на тот же интервал возвращает `409 slot_unavailable`, а пользователь видит понятную ошибку. Backend-тест проверяет код и тело ошибки. Playwright покрывает один conflict path, если его можно сделать устойчивым без хрупкого выбора времени.

### Сценарий 6. Пересекающиеся бронирования запрещены

Нельзя создать бронирование, если новый интервал полностью или частично пересекается с существующим. Ограничение действует даже для разных типов встреч. Сценарий покрывается backend-интеграционными тестами, потому что это доменное правило лучше проверять на API-уровне.

### Сценарий 7. Недопустимое время бронирования отклоняется

Backend отклоняет бронирования в прошлом, вне рабочего окна, за пределами 14-дневного горизонта и с длительностью, не соответствующей выбранному типу встречи. Сценарий покрывается backend-интеграционными тестами.

### Сценарий 8. Владелец создает новый тип встречи

Владелец открывает `/owner/event-types`, заполняет форму создания, сохраняет тип встречи и видит его в списке. Затем гость может увидеть этот тип на главной странице и открыть страницу выбора времени. Сценарий покрывается backend-тестом `POST /owner/event-types` и Playwright e2e.

## Backend Integration Tests

Существующий файл `server/src/__tests__/backend.test.ts` остается основным местом backend-интеграций. Тесты используют:

- `buildApp({ now })` для фиксированного времени;
- `app.inject()` для HTTP-запросов без запуска сетевого сервера;
- `createStorage()` для точечной подготовки состояния, когда нужны уже существующие бронирования.

Новые или уточненные проверки:

- `returns guest event types`;
- `returns available slots for an event type`;
- `creates a booking and exposes it to owner bookings`;
- `removes or marks booked slot unavailable after booking`;
- `rejects an already booked slot`;
- `rejects partially overlapping bookings`;
- `rejects overlap across different event types`;
- `rejects booking outside the working window`;
- `rejects booking for a past slot on the current day`;
- `rejects booking beyond the 14-day availability horizon`;
- `rejects booking with range duration different from event type duration`;
- `rejects invalid guest payloads with 400`.

## Playwright E2E Tests

Добавляется Playwright с конфигурацией `playwright.config.ts`. Тесты используют Chromium как базовый браузер для CI. Desktop viewport достаточен для первого этапа, потому что цель e2e здесь - проверить пользовательский поток и интеграцию frontend с backend, а не адаптивную верстку.

Минимальный набор:

1. `guest can book a consultation and owner sees it`
   - открыть `/`;
   - выбрать `Консультация`;
   - выбрать первый доступный слот;
   - заполнить имя и email;
   - нажать `Забронировать`;
   - проверить подтверждение;
   - открыть `/owner/bookings`;
   - проверить имя, email, тип встречи и статус.

2. `owner can create event type and guest can select it`
   - открыть `/owner/event-types`;
   - заполнить форму создания;
   - сохранить тип встречи;
   - проверить появление в списке владельца;
   - открыть `/`;
   - проверить появление нового типа у гостя;
   - открыть страницу выбора времени.

3. `guest sees conflict error for unavailable slot`
   - подготовить занятый слот через API или тестовый setup;
   - попытаться занять тот же слот через UI, если flow можно сделать стабильным;
   - проверить сообщение `Выбранный слот уже недоступен.`;
   - если UI-вариант получается хрупким, оставить этот сценарий на backend-уровне и не дублировать его в e2e.

Для устойчивости селекторов предпочтение отдается ролям и доступным именам. Если существующих доступных имен недостаточно, допускается точечно добавить `data-testid` для слотов и ключевых контейнеров.

## CI

Добавляется отдельный workflow `.github/workflows/test.yml`. Служебный `.github/workflows/hexlet-check.yml` не изменяется.

CI запускается на push и pull request. Базовые шаги:

1. `actions/checkout`.
2. `actions/setup-node` с Node.js 22.
3. `npm ci`.
4. `npx playwright install --with-deps chromium`.
5. `npm run backend:build`.
6. `npm run backend:test`.
7. `npm run build`.
8. `npm run e2e`.

Playwright поднимает сервисы через `webServer`:

- backend: `npm run backend:dev`, порт `4010`;
- frontend: `npm run dev -- --host 127.0.0.1`, порт `5173`;
- frontend получает `VITE_API_BASE_URL=http://127.0.0.1:4010`.

## Package Scripts

В `package.json` добавляются команды:

- `test:integration`: запуск backend-интеграционных тестов;
- `e2e`: запуск Playwright;
- `test`: последовательный запуск integration и e2e тестов.

Команда `backend:test` сохраняется для совместимости с README и текущим workflow разработчика.

## Риски и смягчение

- Playwright-тесты могут стать хрупкими из-за выбора конкретного времени. Смягчение: выбирать первый доступный слот по UI или через API setup, не завязываться на локализованный формат даты.
- In-memory storage сбрасывается между процессами. Смягчение: каждый e2e запускает чистый backend через `webServer`, а тесты не должны зависеть друг от друга.
- Frontend local state после создания типа встречи может отличаться от persisted state. Смягчение: e2e должен проверять сценарий в рамках одного backend-процесса и учитывать фактическое API-поведение.
- CI станет дольше из-за установки браузера. Смягчение: на первом этапе использовать только Chromium.

## Критерии готовности

- `docs/test-scenarios.md` отражает основные сценарии и уровень автоматизации.
- Backend-интеграционные тесты покрывают happy path, конфликты, пересечения, временные ограничения и валидацию.
- Playwright e2e покрывает успешное бронирование, просмотр бронирования владельцем и создание типа встречи владельцем.
- CI workflow запускает backend build, backend tests, frontend build и e2e.
- Локально проходят `npm run backend:test`, `npm run build` и `npm run e2e`.
