# Сценарии тестирования

Документ фиксирует ключевые пользовательские fullstack-сценарии для проверки приложения «Запись на звонок». Эти сценарии используются как основа для ручной проверки, интеграционных тестов backend и e2e-тестов frontend через Playwright.

## Роли

- `Гость` просматривает типы встреч, выбирает слот и создает бронирование.
- `Владелец` управляет типами встреч и просматривает предстоящие бронирования.

## Сценарий 1. Просмотр доступных типов встреч

**Цель:** убедиться, что гость видит список доступных типов встреч и их основные параметры.

**Предусловия:**
- backend запущен;
- frontend запущен;
- в системе доступны seed-типы встреч `consultation` и `demo`.

**Шаги:**
1. Открыть главную страницу приложения.
2. Перейти к списку типов встреч.
3. Проверить, что в списке отображаются карточки встреч.

**Ожидаемый результат:**
- список типов встреч успешно загружается;
- для каждой встречи отображаются название, описание и длительность;
- гость может перейти к просмотру доступных слотов.

## Сценарий 2. Создание нового типа встречи владельцем

**Цель:** убедиться, что владелец может создать новый тип встречи через интерфейс.

**Предусловия:**
- backend запущен;
- frontend запущен;
- страница управления типами встреч доступна.

**Шаги:**
1. Открыть страницу владельца с управлением типами встреч.
2. Нажать кнопку создания нового типа встречи.
3. Заполнить название, описание и длительность.
4. Сохранить новую запись.

**Ожидаемый результат:**
- новый тип встречи успешно создается;
- созданная запись появляется в списке типов встреч;
- карточка доступна для выбора на гостевой части приложения.

## Сценарий 3. Успешное бронирование встречи гостем

**Цель:** убедиться, что гость может выбрать свободный слот и создать бронирование.

**Предусловия:**
- backend запущен;
- frontend запущен;
- в системе есть хотя бы один тип встречи с доступными слотами.

**Шаги:**
1. Открыть гостевую страницу со списком типов встреч.
2. Выбрать тип встречи.
3. Перейти к списку доступных слотов.
4. Выбрать свободный слот.
5. Заполнить имя и email.
6. Подтвердить бронирование.

**Ожидаемый результат:**
- бронирование успешно создается;
- пользователь видит подтверждение успешной записи;
- выбранный слот становится недоступным для повторного бронирования.

## Сценарий 4. Отображение бронирования у владельца

**Цель:** убедиться, что созданное гостем бронирование отображается в списке владельца.

**Предусловия:**
- backend запущен;
- frontend запущен;
- в системе уже создано хотя бы одно бронирование.

**Шаги:**
1. Открыть страницу владельца со списком бронирований.
2. Дождаться загрузки данных.
3. Найти только что созданную запись.

**Ожидаемый результат:**
- в списке отображается созданное бронирование;
- у записи корректно показаны имя гостя, email, тип встречи, дата и время;
- в списке отображаются только предстоящие подтвержденные бронирования.

## Сценарий 5. Ошибка при повторном бронировании занятого слота

**Цель:** убедиться, что система не позволяет повторно занять уже забронированный слот.

**Предусловия:**
- backend запущен;
- frontend запущен;
- для выбранного типа встречи уже существует бронирование на конкретный слот.

**Шаги:**
1. Открыть гостевую страницу нужного типа встречи.
2. Попробовать заново отправить бронирование на уже занятый слот.
3. Дождаться ответа системы.

**Ожидаемый результат:**
- создание второго бронирования не выполняется;
- пользователь видит понятное сообщение об ошибке;
- backend возвращает конфликт бронирования с кодом `409 slot_unavailable`.

## Область покрытия

Эти сценарии служат основой для двух уровней автоматизации:

- `Vitest` проверяет backend-интеграции и бизнес-правила API.
- `Playwright` проверяет ключевые пользовательские потоки через браузер.
