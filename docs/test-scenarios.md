# Сценарии тестирования

Документ фиксирует ключевые пользовательские fullstack-сценарии приложения «Запись на звонок» и связывает их с уровнями автоматизации.

## Уровни покрытия

- `Backend integration` — Vitest + Fastify `app.inject()`. Проверяет API, бизнес-правила бронирования, конфликты слотов, временные ограничения и валидацию.
- `Playwright e2e` — браузерные тесты ключевых пользовательских потоков через frontend и реальный backend.
- `Manual/optional` — сценарии, которые можно проверить вручную или добавить позже, если они станут критичными.

## Роли

- `Гость` просматривает типы встреч, выбирает слот и создает бронирование.
- `Владелец` управляет типами встреч и просматривает предстоящие бронирования.

## Сценарий 1. Просмотр доступных типов встреч

**Цель:** убедиться, что гость видит список доступных типов встреч и их основные параметры.

**Предусловия:**
- backend запущен;
- frontend запущен для e2e-проверки;
- в системе доступны seed-типы встреч `consultation` и `demo`.

**Шаги:**
1. Открыть главную страницу приложения.
2. Проверить, что в списке отображаются карточки встреч.
3. Проверить название, описание и длительность.
4. Перейти к просмотру доступных слотов.

**Ожидаемый результат:**
- список типов встреч успешно загружается;
- для каждой встречи отображаются название, описание и длительность;
- гость может перейти к странице выбора слота.

**Автоматизация:**
- `Backend integration`: `GET /event-types`.
- `Playwright e2e`: главная страница и переход к странице встречи.

## Сценарий 2. Создание нового типа встречи владельцем

**Цель:** убедиться, что владелец может создать новый тип встречи, а гость может выбрать его.

**Предусловия:**
- backend запущен;
- frontend запущен для e2e-проверки;
- страница управления типами встреч доступна.

**Шаги:**
1. Открыть `/owner/event-types`.
2. Заполнить название, описание и длительность.
3. Сохранить новую запись.
4. Проверить, что запись появилась в списке владельца.
5. Открыть главную страницу гостя.
6. Проверить, что новый тип встречи доступен для выбора.

**Ожидаемый результат:**
- новый тип встречи успешно создается;
- созданная запись появляется в списке владельца;
- карточка доступна для выбора на гостевой части приложения.

**Автоматизация:**
- `Backend integration`: `POST /owner/event-types`.
- `Playwright e2e`: создание типа встречи и переход к нему из гостевой части.

## Сценарий 3. Успешное бронирование встречи гостем

**Цель:** убедиться, что гость может выбрать свободный слот и создать бронирование.

**Предусловия:**
- backend запущен;
- frontend запущен для e2e-проверки;
- в системе есть хотя бы один тип встречи с доступными слотами.

**Шаги:**
1. Открыть гостевую страницу со списком типов встреч.
2. Выбрать тип встречи.
3. Перейти к списку доступных слотов.
4. Выбрать свободный слот.
5. Заполнить имя и email.
6. Подтвердить бронирование.

**Ожидаемый результат:**
- бронирование успешно создается;
- пользователь видит подтверждение успешной записи;
- выбранный слот становится недоступным для повторного бронирования.

**Автоматизация:**
- `Backend integration`: `POST /bookings`, повторная проверка доступных слотов.
- `Playwright e2e`: happy path гостевого бронирования.

## Сценарий 4. Отображение бронирования у владельца

**Цель:** убедиться, что созданное гостем бронирование отображается в списке владельца.

**Предусловия:**
- backend запущен;
- frontend запущен для e2e-проверки;
- в системе уже создано хотя бы одно бронирование.

**Шаги:**
1. Открыть `/owner/bookings`.
2. Дождаться загрузки данных.
3. Найти только что созданную запись.

**Ожидаемый результат:**
- в списке отображается созданное бронирование;
- у записи корректно показаны имя гостя, email, тип встречи, дата, время и статус;
- в списке отображаются только предстоящие подтвержденные бронирования.

**Автоматизация:**
- `Backend integration`: `GET /owner/bookings`.
- `Playwright e2e`: проверка списка владельца после успешного бронирования.

## Сценарий 5. Ошибка при повторном бронировании занятого слота

**Цель:** убедиться, что система не позволяет повторно занять уже забронированный слот.

**Предусловия:**
- для выбранного типа встречи уже существует бронирование на конкретный слот.

**Шаги:**
1. Попробовать повторно создать бронирование на тот же интервал.
2. Дождаться ответа системы.

**Ожидаемый результат:**
- создание второго бронирования не выполняется;
- backend возвращает `409 slot_unavailable`;
- пользователь видит понятное сообщение об ошибке.

**Автоматизация:**
- `Backend integration`: обязательное покрытие.
- `Playwright e2e`: optional, если сценарий можно реализовать устойчиво без хрупкой привязки к дате и порядку слотов.

## Сценарий 6. Ошибка при пересекающемся бронировании

**Цель:** убедиться, что система запрещает полное или частичное пересечение интервалов.

**Предусловия:**
- в системе уже есть подтвержденное бронирование.

**Шаги:**
1. Создать новое бронирование, которое частично пересекается с существующим.
2. Повторить проверку для другого типа встречи, если длительность позволяет пересечение.

**Ожидаемый результат:**
- пересекающееся бронирование отклоняется;
- ограничение действует между разными типами встреч;
- backend возвращает `409 slot_unavailable`.

**Автоматизация:**
- `Backend integration`: обязательное покрытие.
- `Playwright e2e`: не требуется на первом этапе.

## Сценарий 7. Ошибка при недопустимом времени бронирования

**Цель:** убедиться, что backend принимает только допустимые интервалы.

**Предусловия:**
- выбран существующий тип встречи.

**Шаги:**
1. Попробовать создать бронирование в прошлом.
2. Попробовать создать бронирование вне рабочего окна 09:00-19:00.
3. Попробовать создать бронирование за пределами ближайших 14 дней.
4. Попробовать создать бронирование с длительностью, отличающейся от длительности типа встречи.

**Ожидаемый результат:**
- недопустимые интервалы отклоняются;
- backend возвращает `409 booking_out_of_range` для времени вне окна;
- backend возвращает `409 slot_unavailable` для интервала неправильной длительности.

**Автоматизация:**
- `Backend integration`: обязательное покрытие.
- `Playwright e2e`: не требуется на первом этапе.

## Сценарий 8. Валидация данных бронирования

**Цель:** убедиться, что backend отклоняет некорректные данные гостя и временного интервала.

**Предусловия:**
- выбран существующий тип встречи.

**Шаги:**
1. Отправить бронирование без имени.
2. Отправить бронирование с некорректным email.
3. Отправить бронирование с некорректным ISO datetime.

**Ожидаемый результат:**
- backend возвращает `400 validation_error`;
- ответ содержит сообщение о некорректности формы.

**Автоматизация:**
- `Backend integration`: обязательное покрытие.
- `Playwright e2e`: не требуется на первом этапе, потому что HTML form validation уже блокирует часть ошибок в браузере.

## Область покрытия

Минимальная автоматизация для текущего этапа:

- `Vitest` проверяет backend-интеграции и бизнес-правила API.
- `Playwright` проверяет основные пользовательские потоки через браузер:
  - успешное бронирование гостем;
  - отображение бронирования у владельца;
  - создание типа встречи владельцем и выбор этого типа гостем.

Конфликты, пересечения и точные временные ограничения остаются на backend-уровне как более стабильные и быстрые проверки.
