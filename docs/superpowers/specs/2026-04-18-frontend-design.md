# Frontend Design: Запись на звонок

Дата: 2026-04-18

## Статус

Утверждено для MVP.

## Цель

Спроектировать frontend для MVP веб-приложения "Запись на звонок" поверх существующего API-контракта в `spec/api.tsp`.

Frontend должен закрывать два сценария:

- гость выбирает тип встречи, выбирает свободный слот и создает бронирование;
- владелец управляет типами встреч и просматривает предстоящие бронирования.

## Зафиксированный стек

- React
- TypeScript
- Vite
- Mantine
- React Router
- TanStack Query
- Prism для mock API во время разработки

Prism используется через OpenAPI-спецификацию, сгенерированную из TypeSpec:

```text
spec/api.tsp -> OpenAPI JSON/YAML -> Prism mock server -> frontend API client
```

## Архитектурный подход

Используем подход `Pages + shared API layer`.

Основная идея:

- страницы соответствуют маршрутам приложения;
- общие UI-компоненты не знают о деталях HTTP;
- API-клиент изолирует работу с REST JSON API;
- TanStack Query hooks инкапсулируют загрузку, ошибки, кеширование и mutations;
- формы и отображение строятся на Mantine;
- глобальный state manager в MVP не нужен.

Этот подход проще полноценной feature-based архитектуры, но сохраняет важную границу между UI и API-контрактом.

## Маршруты

Гостевая часть:

```text
/                         список типов встреч
/event-types/:id          детали типа встречи, доступные слоты, форма бронирования
```

Раздел владельца:

```text
/owner/event-types        список типов встреч, создание и редактирование
/owner/bookings           список предстоящих бронирований
```

Авторизация не добавляется. Владелец считается заранее заданным, как описано в MVP.

Ожидаемая структура после scaffold:

```text
src/
```

Имена файлов могут уточняться при реализации, но границы слоев должны сохраняться.

## API-слой

API-слой должен опираться на актуальный контракт `spec/api.tsp`.

Минимальный набор frontend-операций:

- `GET /event-types`
- `GET /event-types/{eventTypeId}`
- `GET /event-types/{eventTypeId}/slots`
- `POST /bookings`
- `GET /owner/event-types`
- `POST /owner/event-types`
- `PATCH /owner/event-types/{eventTypeId}`
- `GET /owner/bookings`

На первом этапе допустим аккуратный ручной API-клиент с TypeScript-типами, синхронизированными с контрактом. Если появится генерация типов из OpenAPI, она должна заменить ручные типы без изменения страниц.

## UI и сценарии

### Гость

Страница `/`:

- показывает список типов встреч;
- отображает длительность и описание;
- ведет на страницу выбранного типа встречи.

Страница `/event-types/:id`:

- показывает данные типа встречи;
- загружает свободные слоты;
- группирует или визуально разделяет слоты по дням;
- позволяет выбрать один слот;
- показывает форму с полями "Имя" и "Email";
- создает бронирование через `POST /bookings`;
- после успеха показывает подтверждение.

### Владелец

Страница `/owner/event-types`:

- показывает список типов встреч;
- позволяет создать тип встречи;
- позволяет редактировать название, описание и длительность;
- не добавляет удаление, так как его нет в MVP-контракте.

Страница `/owner/bookings`:

- показывает предстоящие бронирования;
- отображает гостя, email, тип встречи, дату, время и статус;
- не добавляет отмену или перенос.

## Работа со временем

API передает даты и время в UTC.

Frontend отвечает за:

- отображение дат и времени в локальном формате пользователя;
- отправку выбранного интервала обратно в API как UTC `startAt` и `endAt`;
- сохранение инварианта `startAt < endAt`.

Расчет доступности остается на backend/API стороне. Frontend не должен самостоятельно вычислять свободные слоты.

## Ошибки и состояния

Каждая страница с серверными данными должна иметь состояния:

- загрузка;
- пустой список;
- ошибка API;
- успешные данные.

`ErrorResponse.message` отображается пользователю как русскоязычное сообщение. `ErrorResponse.code` используется только для технического ветвления, если оно понадобится.

Для mutations нужны понятные реакции:

- создание бронирования: успех, ошибка недоступного слота, ошибка валидации;
- создание/редактирование типа встречи: успех, ошибка валидации, общая ошибка.

## Тестовые ориентиры

Минимально проверить:

- рендер списка типов встреч;
- переход к странице выбранного типа встречи;
- выбор слота и отправку формы бронирования;
- отображение API-ошибки на русском языке;
- создание и редактирование типа встречи владельцем;
- отображение списка предстоящих бронирований;
- работу frontend с Prism mock server по OpenAPI.

## Вне scope

В frontend MVP не входят:

- авторизация;
- регистрация;
- несколько владельцев;
- удаление типов встреч;
- отмена и перенос бронирований;
- email-уведомления;
- интеграции с календарями;
- глобальный state manager;
- кастомная дизайн-система поверх Mantine.
