# Frontend Implementation Plan: Запись на звонок

Дата: 2026-04-18

Основано на design doc:

- `docs/superpowers/specs/2026-04-18-frontend-design.md`

## Цель

Реализовать frontend MVP на React, TypeScript, Vite и Mantine, сохраняя API-контракт `spec/api.tsp` источником правды.

План разбит на инкременты. Каждый инкремент должен завершаться проверяемым состоянием приложения.

## Инкремент 1: Scaffold и базовая инфраструктура

Задачи:

- создать Vite React TypeScript приложение в корне проекта или согласованной frontend-директории;
- установить Mantine, React Router и TanStack Query;
- настроить `App`, router и providers;
- добавить базовый layout приложения;
- добавить общие состояния `LoadingState`, `ErrorState`, `EmptyState`;
- добавить `.env.example` с переменной базового URL API.

Ожидаемый результат:

- dev server запускается;
- приложение открывается в браузере;
- есть базовая навигация между гостевой частью и разделом владельца.

Проверка:

- `npm run dev`;
- `npm run build`;
- ручная проверка маршрутов.

## Инкремент 2: OpenAPI и Prism flow

Задачи:

- настроить генерацию OpenAPI из `spec/api.tsp`;
- добавить npm script для генерации OpenAPI;
- добавить npm script для запуска Prism mock server;
- зафиксировать путь к сгенерированной OpenAPI-спецификации;
- проверить, что Prism отвечает по MVP endpoint'ам.

Ожидаемый результат:

- frontend можно разрабатывать против mock API без готового backend;
- API flow остается привязанным к TypeSpec-контракту.

Проверка:

- команда генерации OpenAPI проходит без ошибок;
- Prism стартует;
- хотя бы один публичный endpoint возвращает mock response.

## Инкремент 3: Shared API layer

Задачи:

- добавить TypeScript-типы для `EventType`, `Booking`, `Slot`, форм и `ErrorResponse`;
- добавить `client.ts` для REST JSON запросов;
- добавить обработку API ошибок с сохранением `ErrorResponse.message`;
- добавить TanStack Query hooks для всех MVP операций;
- вынести форматирование даты и времени в `shared/lib/dateTime.ts`.

Ожидаемый результат:

- страницы не вызывают `fetch` напрямую;
- запросы и mutations доступны через hooks;
- русскоязычные сообщения ошибок можно показывать в UI.

Проверка:

- build проходит;
- API hooks типизированы;
- ошибки Prism или mock responses корректно превращаются в UI-состояния.

## Инкремент 4: Гостевой путь

Задачи:

- реализовать страницу `/` со списком типов встреч;
- реализовать страницу `/event-types/:id`;
- показать детали типа встречи;
- загрузить и отобразить слоты;
- визуально разделить слоты по дням;
- добавить выбор слота;
- добавить форму бронирования с полями "Имя" и "Email";
- отправлять `POST /bookings`;
- показывать подтверждение после успешного бронирования;
- показывать загрузку, пустой список и API ошибки.

Ожидаемый результат:

- гость может пройти весь путь от списка типов встреч до подтверждения бронирования.

Проверка:

- ручной сценарий гостя против Prism;
- проверка ошибки недоступного слота или валидации, если Prism/mock это позволяет;
- `npm run build`.

## Инкремент 5: Раздел владельца

Задачи:

- реализовать страницу `/owner/event-types`;
- показать список типов встреч;
- добавить форму создания типа встречи;
- добавить редактирование названия, описания и длительности;
- реализовать страницу `/owner/bookings`;
- показать список предстоящих бронирований;
- отображать гостя, email, тип встречи, дату, время и статус;
- показывать загрузку, пустой список и API ошибки.

Ожидаемый результат:

- владелец может управлять типами встреч и видеть предстоящие бронирования без авторизации.

Проверка:

- ручной сценарий владельца против Prism;
- создание и редактирование типа встречи;
- `npm run build`.

## Инкремент 6: UI polish и приемочная проверка

Задачи:

- пройти все страницы на desktop и mobile viewport;
- проверить, что тексты не вылезают из кнопок, карточек и таблиц;
- проверить пустые состояния;
- проверить сообщения ошибок на русском языке;
- проверить отображение UTC времени в локальном формате;
- убрать временные mock helpers, если они не нужны;
- обновить README командами запуска frontend, OpenAPI generation и Prism.

Ожидаемый результат:

- frontend MVP выглядит цельно и готов к подключению реального backend;
- README содержит понятные команды запуска.

Проверка:

- `npm run build`;
- ручная проверка гостевого пути;
- ручная проверка раздела владельца;
- проверка Prism flow.

## Предлагаемые npm scripts

Точные команды зависят от выбранных пакетов после scaffold, но ожидаемый набор такой:

```text

npm run build
npm run preview
npm run api:openapi
npm run api:mock
```

## Порядок реализации

1. Scaffold и базовая инфраструктура.
2. OpenAPI и Prism flow.
3. Shared API layer.
4. Гостевой путь.
5. Раздел владельца.
6. UI polish и приемочная проверка.

## Ограничения

Не добавлять в рамках этого плана:

- авторизацию;
- регистрацию;
- удаление типов встреч;
- отмену или перенос бронирований;
- несколько владельцев;
- интеграции с календарями;
- email-уведомления;
- глобальный state manager;
- кастомную дизайн-систему поверх Mantine.

## Риски и решения

Риск: TypeSpec может потребовать дополнительной настройки генерации OpenAPI.

Решение: сначала проверить генерацию OpenAPI отдельным инкрементом, до активной реализации страниц.

Риск: Prism mock responses могут быть слишком схематичными для удобной ручной проверки.

Решение: при необходимости добавить OpenAPI examples в контракт, не меняя поведение API.

Риск: ручные TypeScript-типы могут разойтись с контрактом.

Решение: держать типы в `shared/api/types.ts`, сверять их с `spec/api.tsp`, а при появлении генерации типов заменить только API-слой без изменения страниц.
