# Implementation Plan: тесты сценариев бронирования

Дата: 2026-04-24

Основано на design doc:

- `docs/superpowers/specs/2026-04-24-booking-tests-design.md`

## Цель

Подготовить проверяемое тестовое покрытие основных сценариев бронирования: зафиксировать сценарии, расширить backend integration tests, добавить Playwright e2e и включить запуск тестов в отдельный GitHub Actions workflow.

План разбит на инкременты. Каждый инкремент должен оставлять проект в рабочем состоянии.

## Инкремент 1: Актуализировать сценарии тестирования

Задачи:

- обновить `docs/test-scenarios.md`;
- явно разделить сценарии по уровням покрытия: backend integration, Playwright e2e, manual/optional;
- добавить сценарии пересекающихся бронирований, конфликта между разными типами встреч, недопустимого времени и создания типа встречи владельцем;
- оставить документ пригодным для ручной проверки и для трассировки к автотестам.

Ожидаемый результат:

- документ описывает основные пользовательские сценарии и показывает, каким уровнем тестов каждый сценарий покрывается;
- будущие тесты можно сверять с одним источником сценариев.

Проверка:

- ручное чтение `docs/test-scenarios.md`;
- отсутствие устаревших утверждений о покрытии.

## Инкремент 2: Расширить backend integration tests

Задачи:

- расширить `server/src/__tests__/backend.test.ts`;
- переиспользовать текущие helpers для фиксированной даты и рабочих слотов;
- добавить helper для произвольного ISO-интервала, если он упростит тесты пересечений;
- проверить, что после бронирования соответствующий слот больше не доступен;
- проверить частичное пересечение слотов;
- проверить конфликт между разными типами встреч;
- проверить бронирование за пределами 14-дневного горизонта;
- проверить несовпадение длительности интервала с длительностью типа встречи;
- проверить невалидные payloads гостя и ожидаемый `400`.

Ожидаемый результат:

- backend-интеграции покрывают happy path, конфликты, временные ограничения и валидацию;
- тесты остаются быстрыми и не требуют сетевого порта.

Проверка:

- `npm run backend:test`;
- `npm run backend:build`.

## Инкремент 3: Добавить Playwright инфраструктуру

Задачи:

- установить Playwright dev dependency;
- добавить `playwright.config.ts`;
- настроить Chromium как основной browser project;
- настроить `webServer` для backend и frontend;
- передать frontend переменную `VITE_API_BASE_URL=http://127.0.0.1:4010`;
- добавить npm scripts `e2e`, `test:integration` и `test`;
- добавить папку для e2e-тестов, например `e2e/`.

Ожидаемый результат:

- Playwright может локально поднять backend и frontend и открыть приложение;
- e2e-команда интегрирована в `package.json`.

Проверка:

- `npx playwright test --list`;
- `npm run build`.

## Инкремент 4: Реализовать Playwright happy path бронирования

Задачи:

- создать e2e-тест `guest can book a consultation and owner sees it`;
- открыть главную страницу;
- выбрать карточку `Консультация`;
- выбрать первый доступный слот;
- заполнить поля `Имя` и `Email`;
- отправить форму бронирования;
- проверить экран подтверждения;
- перейти на `/owner/bookings`;
- проверить, что владелец видит имя, email, тип встречи и статус.

Ожидаемый результат:

- основной пользовательский поток гостя и владельца проверяется через браузер;
- тест не зависит от конкретной даты в UI.

Проверка:

- `npm run e2e`.

## Инкремент 5: Реализовать Playwright flow владельца

Задачи:

- создать e2e-тест `owner can create event type and guest can select it`;
- открыть `/owner/event-types`;
- заполнить название, описание и длительность;
- сохранить новый тип встречи;
- проверить появление записи в списке владельца;
- открыть `/`;
- проверить появление нового типа встречи у гостя;
- открыть страницу выбора времени для созданного типа.

Ожидаемый результат:

- e2e покрывает создание типа встречи владельцем и доступность этого типа для гостевого выбора;
- сценарий проверяет реальную связку frontend/backend в одном процессе CI.

Проверка:

- `npm run e2e`.

## Инкремент 6: Добавить стабильный conflict path

Задачи:

- оценить текущий UI и API setup для проверки конфликта занятого слота через Playwright;
- если сценарий получается стабильным, добавить e2e-тест ошибки `Выбранный слот уже недоступен.`;
- если UI-сценарий требует хрупкого выбора времени, оставить conflict path только на backend-уровне и зафиксировать это в `docs/test-scenarios.md`;
- при необходимости точечно добавить `data-testid` для слотов, не меняя визуальное поведение.

Ожидаемый результат:

- конфликт занятого слота покрыт минимум backend-интеграцией;
- e2e-слой не получает хрупкий тест, который будет падать из-за форматирования даты или порядка слотов.

Проверка:

- `npm run backend:test`;
- `npm run e2e`, если e2e conflict path добавлен.

## Инкремент 7: Настроить GitHub Actions

Задачи:

- добавить `.github/workflows/test.yml`;
- запускать workflow на `push` и `pull_request`;
- использовать Node.js 22;
- установить зависимости через `npm ci`;

- локальный порядок проверок совпадает с CI.

Проверка:

- локально проверить синтаксис YAML визуально;
- после push убедиться, что GitHub Actions workflow проходит.

## Порядок реализации

1. Актуализировать `docs/test-scenarios.md`.
2. Расширить backend integration tests.
3. Добавить Playwright инфраструктуру.
4. Реализовать e2e happy path бронирования.
5. Реализовать e2e flow владельца.
6. Добавить или явно оставить на backend-уровне conflict path.
7. Настроить GitHub Actions.
8. Обновить README и выполнить финальную проверку.

## Предлагаемые npm scripts

```text
npm run backend:test
npm run test:integration
npm run e2e
npm run test
```

Где:

- `backend:test` остается текущей командой Vitest для backend;
- `test:integration` является понятным alias для backend-интеграций;
- `e2e` запускает Playwright;
- `test` запускает integration и e2e последовательно.

## Ограничения

Не добавлять в рамках этого плана:

- авторизацию;
- регистрацию;
- внешнюю базу данных;
- мок календаря;
- email-уведомления;
- визуальное регрессионное тестирование;
- e2e-проверку всех адаптивных breakpoint'ов.

## Риски и решения

Риск: Playwright-тесты станут зависеть от локализованного формата даты.

Решение: выбирать первый доступный слот по роли или `data-testid`, а в проверках использовать имя, email, тип встречи и статус.

Риск: frontend dev server или backend не успеют подняться в CI.

Решение: использовать `webServer.url`, `timeout` и `reuseExistingServer: !process.env.CI` в `playwright.config.ts`.

Риск: установка Playwright browsers сделает CI заметно медленнее.

Решение: установить только Chromium и держать e2e-набор коротким.

Риск: conflict path через UI окажется нестабильным.

Решение: оставить этот сценарий на backend-уровне и документировать решение вместо добавления хрупкого e2e.
