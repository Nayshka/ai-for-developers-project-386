# AGENT.md

## О проекте

Проект: веб-приложение «Запись на звонок».

Цель MVP:
- гость выбирает тип встречи;
- гость видит доступные слоты;
- гость создаёт бронирование;
- владелец управляет типами встреч;
- владелец просматривает список предстоящих встреч.

Проект учебный. Приоритет:
1. понятная архитектура;
2. чистый API-контракт;
3. предсказуемая реализация;
4. минимально достаточный функционал без лишних расширений.

## Границы MVP

В текущий MVP входят:
- список типов встреч;
- просмотр одного типа встречи;
- получение доступных слотов;
- создание бронирования;
- создание и редактирование типов встреч владельцем;
- просмотр предстоящих бронирований владельцем.

В текущий MVP не входят:
- авторизация и регистрация;
- личные кабинеты;
- несколько владельцев;
- интеграции с Google Calendar / Outlook;
- email-уведомления;
- отмена и перенос встреч;
- удаление бронирований;
- сложные workflow сверх минимально нужных.

## Основные сущности

Основные доменные сущности:
- `EventType`
- `Booking`
- `Slot`

Вспомогательные модели:
- `TimeRange`
- `GuestInfo`
- `EventTypeSnapshot`
- `CreateEventTypeForm`
- `UpdateEventTypeForm`
- `CreateBookingForm`
- `ErrorResponse`

## Доменные правила

- Слоты доступны только на ближайшие 14 дней.
- Рабочее окно: ежедневно с `09:00` до `19:00`.
- Нельзя создать бронирование, если его интервал полностью или частично пересекается с существующим активным бронированием.
- Ограничение пересечения действует для всех типов встреч.
- Длительность бронирования должна совпадать с длительностью выбранного типа встречи.
- `Slot` является вычисляемой моделью и не должен рассматриваться как обязательная сущность хранения.
- Один владелец календаря считается захардкоженным на уровне MVP.

## Правила API

Формат API:
- `REST JSON`

Язык пользовательских сообщений:
- сообщения для UI и `ErrorResponse.message` должны быть на русском языке;
- машинные коды ошибок (`ErrorCode`) должны оставаться на английском.

Правила времени:
- все даты и время в API передаются и хранятся в `UTC`;
- локализация и отображение локального времени относятся к frontend-слою;
- для временных интервалов действует правило: `startAt < endAt`.

## Правила моделирования

- Доменные сущности и формы ввода должны быть разделены.
- Для create/update-операций использовать отдельные формы, а не response-модели напрямую.
- В `Booking` хранить снапшот типа встречи (`EventTypeSnapshot`), чтобы история бронирований не менялась после редактирования `EventType`.
- Не добавлять сущности и поля “на будущее”, если они не нужны для текущего MVP.
- Любое расширение контракта должно сначала появиться в спецификации, и только потом в коде.

## Текущий набор endpoint'ов MVP

Публичные:
- `GET /event-types`
- `GET /event-types/{eventTypeId}`
- `GET /event-types/{eventTypeId}/slots`
- `POST /bookings`

Для владельца:
- `GET /owner/event-types`
- `POST /owner/event-types`
- `PATCH /owner/event-types/{eventTypeId}`
- `GET /owner/bookings`

## Frontend Stack

Для frontend MVP используется:
- React;
- TypeScript;
- Vite;
- Mantine;
- React Router;
- TanStack Query;
- Prism для mock API во время разработки.

Архитектурный подход frontend:
- `Pages + shared API layer`.

Prism должен работать от OpenAPI-спецификации, сгенерированной из `spec/api.tsp`.

Во frontend MVP не добавлять:
- глобальный state manager без необходимости;
- авторизацию;
- удаление типов встреч;
- отмену или перенос бронирований;
- кастомную дизайн-систему поверх Mantine.

## Правила изменений

При изменении функциональности придерживаться порядка:
1. обновить `README.md`, если меняется scope или поведение;
2. обновить `spec/api.tsp`;
3. только после этого менять backend;
4. затем адаптировать frontend;
5. проверить, что сообщения об ошибках и пользовательские тексты остались на русском языке.

Не добавлять новые endpoint'ы, статусы, поля и сценарии без явной причины в рамках MVP.

## Правила тестирования

Минимально проверять:
- валидацию входных данных;
- недопустимость пересечения бронирований;
- ограничение окна доступности 14 дней;
- ограничение рабочего времени `09:00-19:00`;
- корректный расчёт слотов для разной длительности встреч;
- создание бронирования для валидного свободного слота.

## Ожидания От Агента

При работе над проектом агент должен:
- опираться на `README.md` и актуальную `TypeSpec`-спецификацию;
- не предлагать функциональность вне MVP без отдельного запроса;
- сохранять простоту архитектуры;
- сначала уточнять или обновлять контракт, если меняется поведение API;
- использовать существующие доменные термины последовательно: `EventType`, `Booking`, `Slot`.
