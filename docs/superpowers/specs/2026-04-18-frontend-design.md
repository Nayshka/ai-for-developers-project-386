# Frontend Design: Запись на звонок

Дата: 2026-04-18

## Статус

Утверждено для MVP.

## Цель

Спроектировать frontend для MVP веб-приложения "Запись на звонок" поверх существующего API-контракта в `spec/api.tsp`.

Frontend должен закрывать два сценария:

- гость выбирает тип встречи, выбирает свободный слот и создает бронирование;
- владелец управляет типами встреч и просматривает предстоящие бронирования.

## Зафиксированный стек

- React
- TypeScript
- Vite
- Mantine
- React Router
- TanStack Query
- Prism для mock API во время разработки

Prism используется через OpenAPI-спецификацию, сгенерированную из TypeSpec:

```text
spec/api.tsp -> OpenAPI JSON/YAML -> Prism mock server -> frontend API client
```

## Архитектурный подход

Используем подход `Pages + shared API layer`.

Основная идея:

- страницы соответствуют маршрутам приложения;
- общие UI-компоненты не знают о деталях HTTP;
- API-клиент изолирует работу с REST JSON API;
- TanStack Query hooks инкапсулируют загрузку, ошибки, кеширование и mutations;
- формы и отображение строятся на Mantine;
- глобальный state manager в MVP не нужен.

Этот подход проще полноценной feature-based архитектуры, но сохраняет важную границу между UI и API-контрактом.

## Маршруты

Гостевая часть:

```text
/                         список типов встреч
/event-types/:id          детали типа встречи, доступные слоты, форма бронирования
```

Раздел владельца:

```text
/owner/event-types        список типов встреч, создание и редактирование
/owner/bookings           список предстоящих бронирований
```

Авторизация не добавляется. Владелец считается заранее заданным, как описано в MVP.

## Структура frontend-кода

Ожидаемая структура после scaffold:

```text
src/
  app/
    App.tsx
    router.tsx
    providers.tsx
  pages/
    GuestEventTypesPage.tsx
    GuestBookingPage.tsx
    OwnerEventTypesPage.tsx
    OwnerBookingsPage.tsx
  shared/
    api/
      client.ts
      types.ts
      queries.ts
    ui/
      AppLayout.tsx
      ErrorState.tsx
      LoadingState.tsx
      EmptyState.tsx
    lib/
      dateTime.ts
```

Имена файлов могут уточняться при реализации, но границы слоев должны сохраняться.

## API-слой

API-слой должен опираться на актуальный контракт `spec/api.tsp`.

Минимальный набор frontend-операций:

- `GET /event-types`
- `GET /event-types/{eventTypeId}`
- `GET /event-types/{eventTypeId}/slots`
- `POST /bookings`
- `GET /owner/event-types`
- `POST /owner/event-types`
- `PATCH /owner/event-types/{eventTypeId}`
- `GET /owner/bookings`

На первом этапе допустим аккуратный ручной API-клиент с TypeScript-типами, синхронизированными с контрактом. Если появится генерация типов из OpenAPI, она должна заменить ручные типы без изменения страниц.

## UI и сценарии

### Гость

Страница `/`:

- показывает список типов встреч;
- отображает длительность и описание;
- ведет на страницу выбранного типа встречи.

Страница `/event-types/:id`:

- показывает данные типа встречи;
- загружает свободные слоты;
- группирует или визуально разделяет слоты по дням;
- позволяет выбрать один слот;
- показывает форму с полями "Имя" и "Email";
- создает бронирование через `POST /bookings`;
- после успеха показывает подтверждение.

### Владелец

Страница `/owner/event-types`:

- показывает список типов встреч;
- позволяет создать тип встречи;
- позволяет редактировать название, описание и длительность;
- не добавляет удаление, так как его нет в MVP-контракте.

Страница `/owner/bookings`:

- показывает предстоящие бронирования;
- отображает гостя, email, тип встречи, дату, время и статус;
- не добавляет отмену или перенос.

## Visual UI decisions

Интерфейс должен выглядеть как спокойное рабочее приложение, а не как маркетинговый landing page.

Общие решения:

- использовать светлый фон страницы и белые рабочие поверхности;
- держать карточки и панели с радиусом до `8px`;
- использовать верхнюю навигацию с разделами "Гость" и "Владелец";
- не вкладывать UI cards друг в друга;
- не использовать декоративные градиентные фоны, орбы или иллюстрации;
- показывать реальные рабочие данные: тип встречи, длительность, дату, время, гостя, email и статус;
- пользовательские тексты и ошибки показывать на русском языке;
- машинные статусы вроде `confirmed` допустимы там, где это напрямую отражает API-модель.

### Guest event types page

Страница `/` должна быть первым рабочим экраном приложения.

Композиция:

- верхняя навигация;
- заголовок с прямым смыслом, например "Выберите удобный формат встречи";
- короткое пояснение сценария;
- компактный блок правил записи: рабочее окно `09:00-19:00`, горизонт `14 дней`, локальное отображение времени;
- карточки типов встреч с названием, описанием, длительностью и кнопкой выбора времени.

Карточки типов встреч являются повторяемыми cards. На desktop они могут идти сеткой, на mobile - одной колонкой.

### Guest booking page

Страница `/event-types/:id` должна совмещать выбор слота и форму бронирования.

Композиция:

- ссылка возврата к списку типов встреч;
- блок деталей выбранного типа встречи;
- слева список свободных слотов, визуально разделенный по дням;
- справа панель бронирования с summary выбранного времени и формой;
- выбранный слот должен быть явно подсвечен;
- недоступный слот, если он показан в UI, должен быть визуально отличим и недоступен для выбора.

Форма содержит только поля:

- "Имя";
- "Email".

После `POST /bookings` на успехе показывается confirmation state.

### Booking success state

После успешного бронирования пользователь должен увидеть отдельное подтверждение.

Содержание:

- явное сообщение "Бронирование создано";
- тип встречи;
- дата и время в локальном формате;
- имя гостя;
- email гостя;
- пояснение, что email-уведомления в MVP не отправляются;
- действие для новой записи или возврата к типам встреч.

Экран не должен обещать email, календарное приглашение или отмену встречи.

### Owner event types page

Страница `/owner/event-types` должна быть рабочим экраном управления типами встреч.

Композиция:

- активный раздел "Владелец";
- вкладки или сегментированный переход между "Типы встреч" и "Бронирования";
- список типов встреч с названием, описанием и длительностью;
- действие "Редактировать";
- форма создания/редактирования рядом со списком на desktop и ниже списка на mobile.

Форма содержит только поля из API-контракта:

- название;
- описание;
- длительность в минутах.

Удаление не показывать, так как endpoint удаления не входит в MVP.

### Owner bookings page

Страница `/owner/bookings` должна быть удобной для быстрого просмотра предстоящих встреч.

Композиция:

- активная вкладка "Бронирования";
- короткая сводка по ближайшим встречам, если она вычисляется из уже загруженного списка и не требует нового API;
- таблица на desktop;
- адаптивный список или stacked rows на mobile.

Для каждой встречи показывать:

- дату и время в локальном формате;
- имя гостя;
- email гостя;
- тип встречи из snapshot;
- длительность из snapshot;
- статус.

Не показывать действия отмены или переноса.

## Работа со временем

API передает даты и время в UTC.

Frontend отвечает за:

- отображение дат и времени в локальном формате пользователя;
- отправку выбранного интервала обратно в API как UTC `startAt` и `endAt`;
- сохранение инварианта `startAt < endAt`.

Расчет доступности остается на backend/API стороне. Frontend не должен самостоятельно вычислять свободные слоты.

## Ошибки и состояния

Каждая страница с серверными данными должна иметь состояния:

- загрузка;
- пустой список;
- ошибка API;
- успешные данные.

`ErrorResponse.message` отображается пользователю как русскоязычное сообщение. `ErrorResponse.code` используется только для технического ветвления, если оно понадобится.

Для mutations нужны понятные реакции:

- создание бронирования: успех, ошибка недоступного слота, ошибка валидации;
- создание/редактирование типа встречи: успех, ошибка валидации, общая ошибка.

## Тестовые ориентиры

Минимально проверить:

- рендер списка типов встреч;
- переход к странице выбранного типа встречи;
- выбор слота и отправку формы бронирования;
- отображение API-ошибки на русском языке;
- создание и редактирование типа встречи владельцем;
- отображение списка предстоящих бронирований;
- работу frontend с Prism mock server по OpenAPI.

## Вне scope

В frontend MVP не входят:

- авторизация;
- регистрация;
- несколько владельцев;
- удаление типов встреч;
- отмена и перенос бронирований;
- email-уведомления;
- интеграции с календарями;
- глобальный state manager;
- кастомная дизайн-система поверх Mantine.
